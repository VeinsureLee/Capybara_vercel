# M4: 旅行模块 (`travel`)

> 管理卡皮巴拉多日旅行的触发、地点选择、每日手记生成、旅途相遇匹配、物品收集和惰性完成，是卡皮外出探索世界的核心逻辑。替代 V1 探索模块。

---

## 模块概览

| 属性 | 说明 |
|------|------|
| 职责 | 旅行触发、地点选择、手记生成、相遇匹配、物品收集、惰性完成 |
| 入口 | `/api/travel` (GET/POST), `/travel` 页面 |
| 依赖 | AI Client, Supabase, Prompts, 地点库 |
| 核心 | 意向关键词 → 地点标签匹配 → 多日旅行 → 每日手记 + 匹配 → 惰性完成 |

---

## 旅行时长分布

| 类型 | 概率 | 时长 | 手记数 | 产出物品 |
|------|-----|------|-------|---------|
| 短途（一天游） | 30% | 1 天 | 1 篇 | 1 件 |
| 中途（2-3 天） | 40% | 2-3 天 | 2-3 篇 | 2-3 件 |
| 长途（4-5 天） | 30% | 4-5 天 | 4-5 篇 | 3-5 件 |

旅行结束后进入 1-2 天休息期（resting），年度平均：出门日约 70%，休息日约 30%。

> **⚠️ 测试阶段说明：** 在开发/测试环境中，所有旅行时长单位以**分钟**代替**天**（例如 1-5 天 → 1-5 分钟，休息期 1-2 天 → 1-2 分钟），以便快速验证完整旅行生命周期。上线时切换回天为单位。

```json
  "visual_keywords": ["红色�的鸟居", "苔藓石灯笼", "林间小径"]

```

### 地点匹配算法

```
1. 从近期对话中提取意向关键词（intent keywords）
   - 最近 20 条对话的 keywords 加权聚合（指数衰减 0.85）
   - 取 top 8 作为意向词

2. 意向词 → 地点标签匹配
   - 计算每个地点的 tags 与意向词的重合度
   - 排除 30 天冷却期内的地点
   - 取 Top-5 候选

3. 从 Top-5 中随机选取 1 个
   - 无明显意向时全库随机（排除冷却期）
```

### 30 天冷却期

同一地点在被访问后 30 天内不会再被选中，避免短期重复。冷却记录存储在 `location_visits` 表中。

## 惰性完成机制

旅行完成不依赖定时任务或 Cron Job，而是在用户查询时惰性检查并完成。与 V1 探索模块相同的 Serverless 友好模式。

```
GET /api/travel 时：
  1. 查询 status='ongoing' 的旅行
  2. 如果 estimated_return < now：
     - 更新 travel.status = 'completed'
     - 更新 capybara.status = 'resting'
     - 设置 capybara.last_travel_completed_at = now
     - 返回 { travel, just_completed: true }
  3. 如果还在进行中：
     - 返回 { travel, just_completed: false, current_day: N }
  4. 如果 capybara.status='resting' 且 rest_end < now：
     - 更新 capybara.status = 'home'
     - 返回 { travel: null, just_rested: true }
  5. 如果无进行中旅行：
     - 返回 { travel: null }
```

优势：零运维成本，无需后台定时任务。

## 全局每日匹配

每天系统对所有旅行中的卡皮做全局配对，按记忆相似度匹配。

```
每日匹配流程：
  1. 获取所有 status='ongoing' 的旅行
  2. 提取每只卡皮的可分享记忆（shareable=true, sensitive=false）
  3. 计算两两记忆相似度（主题标签 Jaccard + 情感向量余弦）
  4. 贪心配对：按相似度从高到低，每只卡皮最多匹配 1 个
  5. 每对被送到同一地点共同旅行
  6. 奇数用户剩 1 人 → 当日无匹配（"今天没遇到特别的人"）
  7. 匹配结果写入当日手记
```

匹配深度由 AI 在手记中自然呈现——不设档位标签，相似度高的相遇有更深的情感共振。

## 手记生成

每天为旅行中的卡皮生成一篇叙事手记。

### 手记内容结构

```json
{
  "travel_id": "uuid",
  "day": 2,
  "date": "2026-04-15",
  "location_name": "京都·下鸭神社附近",
  "narrative": "AI 生成的当日叙事文本（80-180字）",
  "encounter": {
    "matched_user_id": "uuid | null",
    "matched_capybara_name": "string | null",
    "encounter_narrative": "相遇段落文本 | null"
  },
  "item_found": {
    "name": "苔藓小石灯",
    "description": "神社步道旁捡到的覆满青苔的小石灯",
    "category": "collectible",
    "rarity": "uncommon"
  } | null,
  "created_at": "timestamp"
}
```

### 手记生成规则

- 每日一篇，由 AI 基于地点描述、卡皮性情、对话历史生成
- 如有匹配相遇，自然织入叙事（不单独推送）
- 物品发现 per day：0-1 件，累积整趟旅行
- 手记末尾如有相遇，附带入口："去 Ta 的河岸看看"

## 旅行触发方式

### 方式 A：对话触发

```
用户聊天 → AI 返回 want_to_travel: true
  → 前端显示旅行提示卡
  → 用户点击"让它去吧"
```

```
用户进入 /travel → 看到出发按钮
  → 点击"让{名字}去旅行"
  → POST /api/travel
```

```
休息期满 → 卡皮自动触发下一次旅行

  → 柔和告知用户
```

### 触发条件检查

1. `capybara.status === 'home'`（必须在家）
2. 无 ongoing 旅行（不能重复）

## REST API

### GET /api/travel

查询当前旅行状态 + 惰性完成检查。

| 返回场景 | 响应 |
|---------|------|
| 旅行进行中 | `{ travel, just_completed: false, current_day: N }` |
| 旅行刚完成 | `{ travel, just_completed: true }` |
| 休息期刚结束 | `{ travel: null, just_rested: true }` |
| 无旅行 | `{ travel: null }` |

### POST /api/travel

发起新旅行。

| 步骤 | 操作 |
|------|------|
| 1 | 验证 capybara.status === 'home' |
| 2 | 聚合意向关键词 → 匹配地点 |
| 3 | 随机旅行时长（1-5 天） |
| 4 | 创建 travel 记录（status: ongoing） |
| 5 | 更新 capybara.status = 'traveling' |
| 6 | 返回 `{ travel, location }` |

## V2.1 探索页面改版

### 改版要点

基于 V2 旅行模块的使用反馈，对探索页面进行以下 5 项改进：

---

### 改进 1：同天内进度条细粒度展示

**问题：** 当前进度条只以"天"为单位，同一天内打开页面多次看到的进度条没有变化，缺乏实时感。

**方案：** 进度条改为基于时间的连续进度，而非离散的天级进度。

```
进度计算公式：
  progress = (now - started_at) / (estimated_return - started_at)

  // 示例：3天旅行，已过1.5天
  // progress = 1.5 / 3 = 50%
```

**进度条 UI 设计：**

```
┌─────────────────────────────────────────────┐
│  📍 京都·下鸭神社附近                         │
│                                             │
│  ██████████████░░░░░░░░░░░░  50%           │
│  ─┬──────┬──────┬──────┬─                   │
│  出发   第1天   第2天   到达                  │
│                ↑ 现在                        │
│                                             │
│  第 2 / 3 天 · 还有 1 天 12 小时             │
│  （今日进度：已过半天）                       │
└─────────────────────────────────────────────┘
```

**实现要点：**
- 进度条为连续填充条，不再是离散色块
- 天数分隔线标注在进度条上方/下方，作为刻度
- 每秒更新 `now`，进度条平滑推进
- 测试模式下同样有效（1分钟=1天，进度条按秒推进）

---

### 改进 2：手记生成阻塞机制

**问题：** 用户点击"生成手记"后，如果 AI 响应较慢，用户可能再次点击。第二次点击时 `dayNumber` 可能已经变化（尤其是测试模式下分钟级切换），导致错误地生成下一天的手记，但 UI 上仍显示为当天的。

**方案：** 前端 + 后端双重阻塞。

**前端阻塞：**
```
1. 点击按钮后立即进入 generatingJournal = true 状态
2. 按钮置灰 + 显示加载动画："正在生成手记..."
3. 请求返回后（成功/失败）才恢复按钮状态
4. 请求期间按钮不可点击（disabled + pointer-events-none）
```

**后端阻塞（POST /api/journal）：**
```
1. 在计算 dayNumber 后，先检查该 dayNumber 是否已有手记（已有此逻辑，返回 409）
2. 新增：在 insert 前加乐观锁，使用 Supabase 的 upsert 或 unique 约束
   - journals 表加 UNIQUE(travel_id, day_number) 约束
   - 即使并发请求也只有一条能成功插入
3. 新增：请求体中传入前端计算的 expected_day，后端校验：
   - 如果后端计算的 dayNumber !== expected_day，返回 conflict 错误
   - 前端收到 conflict 后刷新旅行状态，重新计算天数
```

**前端状态流：**
```
用户点击 → [按钮禁用 + loading] → POST /api/journal(expected_day=N)
  ├── 成功 → 手记加入列表 → [按钮恢复]
  ├── 409 已存在 → 提示"今日手记已生成" → [按钮恢复]
  ├── conflict 天数不一致 → 刷新旅行状态 → [按钮恢复 + 重新显示正确天数]
  └── 网络错误 → 提示错误 → [按钮恢复]
```

```
  "type": "journal_highlight" | "item_found" | "custom",
  "keyword": "绿叶",                    // 从手记中提取的关键词
  "label": "林间步道上落满了银杏叶",      // 简短描述
  "position": { "x": 0.3, "y": 0.45 }, // 图片上的相对坐标 (0-1)
  "link_target": "/journal/{journal_id}" // 点击跳转目标
}
```

**标注展示规则：**
1. 图片上显示圆点标记（●），从圆点拉出一条线连接到对话框
2. 对话框中显示：关键词/物品名 + 简短描述（1行）
3. 点击对话框可跳转到对应的手记详情
4. 标注不超过 5 个，避免图片过于拥挤
5. 默认折叠，点击图片后展开标注

**标注来源（特殊判定逻辑待定，初步方案）：**
- **手记关键词标注：** AI 生成手记时额外返回 `visual_highlights` 字段，包含 1-2 个适合在图片上标注的关键词 + 描述
- **物品标注：** 在该地点发现的 rare/legendary 物品自动生成标注
- **用户自定义标注（后期）：** 用户可以手动在图片上添加标注

**手记生成 prompt 扩展：**
```json
// AI 返回新增字段
{
  "narrative": "...",
  "encounter_narrative": "...",
  "daily_item": {...},
  "visual_highlights": [
    {
      "keyword": "绿叶",
      "description": "林间步道上落满了银杏叶",
      "suggested_position": "left-center"  // AI 建议的大致位置区域
    }
  ]
}
```

**图片详情交互流程：**
```
地图点击地点 → LocationDetail（底部面板，现有）
  → 点击图片区域 → 进入大图详情视图（新页面/模态框）
    → 大图 + 标注展开
    → 点击标注对话框 → 跳转到对应手记详情
    → 双指缩放浏览大图
```

## V1 探索模块向后兼容

V1 探索模块（`/api/explore`）保留不删除，供向后兼容使用。V1 探索为短时单次探索（3-12 分钟），V2 旅行为多日真实地点旅行。

| 维度 | V1 探索 | V2 旅行 |
|------|--------|--------|
| 时长 | 3-12 分钟 | 1-5 天 |
| 地点 | AI 虚构 | 真实世界地点库 |
| 产出 | 单次故事 + 物品 | 每日手记 + 累积物品 |
| 匹配 | 无 | 全局每日匹配 |
| 状态 | exploring | traveling → resting |
| API | /api/explore | /api/travel |
