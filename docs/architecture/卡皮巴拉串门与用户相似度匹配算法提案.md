# 卡皮巴拉串门与用户相似度匹配算法 · 提案

> 面向现有 `visiting` 状态（M2 卡皮巴拉模块）未实现部分的落地设计。
> 目标：让两只属于不同用户的卡皮巴拉互相"交流"，并基于交流结果计算一份用户之间的相似度分数，用于后续"推荐串门对象 / 推荐朋友 / 话题冷启动"等产品能力。

---

## 一、设计目标

1. **不破坏现有数据模型**：复用 `capybaras.traits / mood / memory`、`conversations.keywords`、`explorations.keywords`，不强制引入新的重量级画像系统。
2. **保持拟人化体验**：相似度不是冷冰冰的数字，而是通过两只卡皮巴拉的一次"串门对话"自然产生。
3. **隐私优先**：用户的原始对话内容不跨用户流转，只有"卡皮巴拉可公开的抽象特征"参与匹配与交流。
4. **算力友好**：候选召回走廉价的向量 / 关键词检索，精排才调用 LLM。

---

## 二、整体流程（一次串门 = 一次相似度测量）

```
           ┌─────────────────────────┐
   触发串门 │ 用户A 的卡皮巴拉 (home) │
           └───────────┬─────────────┘
                       │ 1. 构建 PersonaCard(A)
                       ▼
           ┌─────────────────────────┐
   召回     │ 候选池（向量 + 过滤）    │
           └───────────┬─────────────┘
                       │ 2. Top-K 候选 PersonaCard(B1..Bk)
                       ▼
           ┌─────────────────────────┐
   精排     │ 相似度打分 sim(A, Bi)   │
           └───────────┬─────────────┘
                       │ 3. 选出 B*
                       ▼
           ┌─────────────────────────┐
   对话     │ 两只卡皮巴拉 N 轮交流    │ ←— LLM 驱动
           └───────────┬─────────────┘
                       │ 4. 对话后抽取 affinity 信号
                       ▼
           ┌─────────────────────────┐
   回写     │ user_affinity 表 + 状态  │
           └─────────────────────────┘
```

## 三、PersonaCard：卡皮巴拉对外的"名片"

PersonaCard 是参与匹配的唯一实体，目的是把"用户隐私原文"和"可对外交流的抽象特征"隔离开。

```typescript
interface PersonaCard {
  capybara_id: string
  owner_id: string                 // 仅系统侧使用，不出现在对外 prompt 中
  name: string
  traits: string[]                 // 来自 capybaras.traits
  mood: string                     // 最新 mood
  level: number
  memory_topics: string[]          // 从 memory 做"脱敏聚类"后的主题词（见 §4.1）
  topic_vector: number[]           // memory_topics 的 embedding 平均（见 §4.2）
  recent_exploration_tags: string[] // 最近 N 次 explorations.keywords 去重
  updated_at: string
}
```

PersonaCard 随 `capybaras` / `conversations` / `explorations` 的写入增量更新，存在一张缓存表 `persona_cards` 中，避免每次匹配都重算。

### 3.1 memory_topics 的脱敏聚类

`capybaras.memory` 里可能包含"失眠 / 想离职 / 和父母吵架"这类敏感片段，不能原样跨用户可见。做两步处理：

1. **黑名单过滤**：一个小词表把明显 PII / 过敏词替换为上位词（"父母吵架" → "家庭关系"）。
2. **聚类归一**：对剩余关键词用预计算 embedding 做 K-means（K=8），每个簇取簇中心最接近的词作为簇名，只暴露簇名。

这样对外能看到的是"家庭关系 / 睡眠 / 远行 / 自我怀疑 / ..."这种主题级颗粒，不再是具体事件。

## 四、相似度计算：三路融合

把相似度拆成三路廉价信号 + 一路昂贵信号，分别用不同阶段：

| 信号 | 来源 | 阶段 | 成本 |
|------|------|------|------|
| `sim_vec` | topic_vector 余弦相似度 | 召回 | 低 |
| `sim_trait` | traits 的 Jaccard | 召回 + 精排 | 低 |
| `sim_tag` | recent_exploration_tags 的加权 Jaccard | 精排 | 低 |
| `sim_chat` | 两只卡皮巴拉 LLM 对话后的亲和度评分 | 事后 | 高 |

### 4.1 召回：`sim_vec + 硬过滤`

用 pgvector 在 `persona_cards.topic_vector` 上做 ANN，取 Top-K=50。过滤：

- 排除 `owner_id = A.owner_id`
- 排除 30 天内已经互相串门过的对（避免疲劳）
- 可选：只召回 `status = 'home'` 的卡皮巴拉（愿意接待）

### 4.2 精排：加权线性组合

对召回的 50 个候选计算：

```
score(A, Bi) = w1 · sim_vec   (cosine)
             + w2 · sim_trait (Jaccard)
             + w3 · sim_tag   (加权 Jaccard，按探索 recency 衰减)
             - w4 · diversity_penalty(A, Bi)
```

默认 `w = (0.5, 0.2, 0.2, 0.1)`，`diversity_penalty` 用于避免总是匹配到"最像自己"的人，让结果略带惊喜。取 Top-1 作为本次串门对象 B*。

### 4.3 事后：`sim_chat` 与真实亲和度

两只卡皮巴拉用 LLM 做 `N = 4~6` 轮对话（见 §5），对话结束后由**裁判 prompt** 输出：

```json
{
  "affinity": 0.0-1.0,
  "shared_topics": ["远行", "失眠"],
  "tone_match": 0.0-1.0,
  "novelty": 0.0-1.0,
  "summary_for_A": "...",
  "summary_for_B": "..."
}
```

最终落地的用户相似度：

```
user_sim(A, B) = α · score(A, B) + β · affinity + γ · tone_match
```

默认 `α=0.3, β=0.5, γ=0.2`。`user_sim` 连同两份 summary 写入 `user_affinity` 表。

## 五、两只卡皮巴拉如何"交流"

> 不是真的两个模型在跑，工程上就是一个 LLM 轮流扮演 A、B。

### 5.1 Prompt 骨架

```
你现在同时扮演两只卡皮巴拉 A 和 B，它们第一次串门见面。
A 的名片: {PersonaCard(A) 去掉 owner_id}
B 的名片: {PersonaCard(B) 去掉 owner_id}
规则:
- 每次输出 1 轮，格式 { "speaker": "A"|"B", "text": "..." }
- 只能基于名片内容，不得杜撰用户隐私
- 共进行 N 轮，之后额外输出 1 个 JSON 评估块（见 §4.3）
```

### 5.2 N 轮停止条件

满足其一即停：

- 达到 `N_max = 6` 轮；
- 连续 2 轮 embedding 相似度 > 0.95（开始重复）；
- 模型输出评估块。

### 5.3 成本与异步化

- 一次串门约消耗 `6 × 2 × ~150 tokens ≈ 1.8k tokens`，放在 Serverless 后台任务（与现有 `exploring` 的惰性检查一致的模式）。
- `capybaras.status` 先置为 `visiting`，任务完成后回 `home` 并写入 `user_affinity` 与两段对话摘要（只对各自 owner 可见）。

## 六、数据模型增量

```sql
-- 名片缓存
create table persona_cards (
  capybara_id uuid primary key references capybaras(id) on delete cascade,
  owner_id    uuid not null,
  traits        text[],
  memory_topics text[],
  recent_tags   text[],
  topic_vector  vector(384),
  updated_at    timestamptz default now()
);
create index on persona_cards using ivfflat (topic_vector vector_cosine_ops);

-- 用户亲和度（对称存储也可，这里用 (min,max) 规范化去重）
create table user_affinity (
  user_low   uuid not null,
  user_high  uuid not null,
  user_sim   numeric(4,3) not null,
  affinity   numeric(4,3),
  shared_topics text[],
  last_visit_at timestamptz default now(),
  primary key (user_low, user_high),
  check (user_low < user_high)
);

-- 单次串门记录
create table visits (
  id uuid primary key default gen_random_uuid(),
  a_capybara uuid references capybaras(id),
  b_capybara uuid references capybaras(id),
  transcript jsonb,      -- N 轮对话
  eval       jsonb,      -- §4.3 的 JSON
  created_at timestamptz default now()
);
```

RLS：`visits.transcript` 与 `user_affinity` 只允许参与者双方读取；`persona_cards` 对匹配服务以 `service_role` 可读。

## 七、触发与节流

- **主动触发**：用户在 `/explore` 页点"让它去串门"，前端 POST `/api/visit`。
- **被动触发**：每日定时任务，为 `mood ∈ {curious, excited}` 的卡皮巴拉自动安排一次串门。
- **节流**：单只卡皮巴拉每日最多 1 次串门；同一对 30 天内不重复；`user_affinity` 高于 0.85 的对降权以留给新人。

## 八、评估与调参

| 指标 | 含义 | 目标 |
|------|------|------|
| Recall@50 (离线) | 人工标注"应当匹配"的对是否在召回里 | ≥ 0.9 |
| Pairwise AUC | 打分函数区分相关对/随机对 | ≥ 0.8 |
| 人工评分 | 对话读起来"像同一类人" | ≥ 4/5 |
| 回访率 | 用户查看串门摘要后 24h 内再次打开 app | 上升 |

权重 `(w, α, β, γ)` 用 **小规模带标注的 A/B 数据** + 贝叶斯优化离线拟合，再线上灰度。

## 九、分阶段落地

| 阶段 | 范围 |
|------|------|
| P0 | `persona_cards` 物化 + `sim_vec + sim_trait` 精排，无 LLM 交流，`user_sim = score` |
| P1 | 上线两只卡皮巴拉 LLM 对话 + affinity 评估 |
| P2 | 引入 `diversity_penalty`、mood 触发、摘要回写用户 |
| P3 | 把 `user_affinity` 暴露给"推荐朋友 / 相似探索"等下游能力 |

---

## 十、风险与取舍

- **LLM 幻觉暴露隐私**：靠 §3.1 脱敏 + prompt 里"只能基于名片"硬约束，并在裁判步骤加一个"是否含具体隐私事件"的检测分支，命中则丢弃 transcript。
- **冷启动**：新用户 memory 太稀疏，topic_vector 信息量低；P0 可退化为纯 `sim_trait` + 随机探索。
- **回音室**：纯相似度会让用户只遇到自己；`diversity_penalty` 与 P2 的 mood 触发是对冲手段。
- **算力**：P1 起才有 LLM 成本；用 Haiku 级小模型跑双角色扮演足够，必要时只在裁判步骤上 Sonnet。
